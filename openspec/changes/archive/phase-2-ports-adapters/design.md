# Design — Phase 2: Ports & Adapters

Wire the tested scheduling domain to Postgres (read-only) and to inbound HTTP. The
`MessagesService` becomes an orchestrator: interpret → load Agenda → `recalculate` →
return a proposed `PlanResponseDto`. Prisma + Neon are LOCKED. Domain stays pure (ADR-0002):
ZERO `@prisma/*` imports under `src/domain/`. No-double-booking enforced at the DB via a raw-SQL
`EXCLUDE USING gist` migration (ADR-0011). Persistence is READ-ONLY this phase; `change_history`
gets schema + port + a fail-loud stub only (writes are Phase 4).

## Quick path (what apply will build, in order)

1. **DB foundation** — `prisma/schema.prisma`, `prisma migrate dev` (generated migration), then a
   hand-edited `--create-only` migration for the exclusion constraint. `PrismaService`. `.env.example`.
2. **Ports + adapter + mappers + fake** — `AgendaRepository` port, `PrismaAgendaRepository` adapter,
   row↔domain mappers, `InMemoryAgendaRepository` fake, `ChangeHistoryRepository` port + fail-loud stub.
3. **Orchestration + DTO** — `PlanResponseDto` + `mapPlanToDto`, rewritten `MessagesService`/controller,
   `AppModule` composition root, e2e rewrite.

These three chunks map 1:1 to the chained PRs the proposal recommends.

## 2. Prisma schema + migrations

### `apps/scheduling/prisma/schema.prisma`

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model WorkingHours {
  id        String   @id @default(uuid()) @db.Uuid
  doctorId  String   @map("doctor_id") @db.Uuid
  day       DateTime @db.Date
  openTime  DateTime @map("open_time") @db.Time(0)
  closeTime DateTime @map("close_time") @db.Time(0)

  @@unique([doctorId, day])
  @@map("working_hours")
}

model Appointment {
  id        String   @id @default(uuid()) @db.Uuid
  doctorId  String   @map("doctor_id") @db.Uuid
  patientId String   @map("patient_id") @db.Uuid
  day       DateTime @db.Date
  startTime DateTime @map("start_time") @db.Time(0)
  endTime   DateTime @map("end_time") @db.Time(0)

  @@index([doctorId, day])
  @@map("appointments")
}

model ChangeHistory {
  id           String   @id @default(uuid()) @db.Uuid
  doctorId     String   @map("doctor_id") @db.Uuid
  occurredAt   DateTime @default(now()) @map("occurred_at") @db.Timestamptz(6)
  rawMessage   String   @map("raw_message")
  intentKind   String   @map("intent_kind") @db.VarChar(50)
  intentParams Json     @map("intent_params") @db.JsonB
  planSnapshot Json     @map("plan_snapshot") @db.JsonB
  applied      Boolean  @default(false)

  @@index([doctorId, occurredAt])
  @@map("change_history")
}
```

Notes:
- `@db.Time(0)` → Postgres `TIME` (no fractional seconds). Prisma maps `TIME` to a JS `Date` whose
  **time-of-day** carries the value (the date part is a placeholder, typically `1970-01-01`). The mapper
  reads `getHours()/getMinutes()` — see §3.
- `@db.Date` → Postgres `DATE`; arrives as a `Date` at UTC midnight.
- `change_history` is created now (schema only). No adapter wired to it this phase.

### `prisma/migrations/<ts>_no_double_booking/migration.sql` (hand-written body)

```sql
-- btree_gist lets gist indexes mix scalar equality (doctor_id, day) with range overlap.
CREATE EXTENSION IF NOT EXISTS btree_gist;

-- No two appointments for the same doctor on the same day may overlap in time.
-- Half-open range [start, end) matches the domain's overlap semantics (adjacent slots OK).
ALTER TABLE "appointments"
  ADD CONSTRAINT no_double_booking
  EXCLUDE USING gist (
    "doctor_id" WITH =,
    "day" WITH =,
    tsrange(
      ('1970-01-01'::date + "start_time"),
      ('1970-01-01'::date + "end_time"),
      '[)'
    ) WITH &&
  );
```

Rationale for `tsrange(date + time, …)`: Postgres has no native `timerange`. Composing a fixed date
with the `TIME` columns yields a `timestamp`, and `tsrange(..., '[)')` gives half-open overlap that
exactly matches `overlaps()` in `time-slot.ts` (adjacent slots do not collide). Apply this constraint as
the FINAL `ADD CONSTRAINT` (after `migrate dev --name init` created the table) so the table already exists.

### Deploy

`pnpm prisma migrate deploy` runs both migrations in order against `DATABASE_URL` (used in CI / prod /
when the user first points at Neon). `migrate dev` is for local schema iteration only.

## 3. Mapping layer (exact functions)

### `infrastructure/prisma/agenda.mapper.ts` — rows → domain

Prisma `TIME` columns arrive as JS `Date` with the value encoded in the **UTC** time-of-day (Prisma
serializes `TIME` against `1970-01-01T00:00:00Z`). Read with `getUTCHours()/getUTCMinutes()` to avoid
the machine-timezone trap.

```ts
import { Appointment as ApptRow, WorkingHours as WhRow } from '@prisma/client';
import {
  agenda, appointment, timeSlot, workingHours, localTime,
} from '../../../domain';
import type { Agenda } from '../../../domain';

/** Postgres TIME (via Prisma) → domain LocalTime. */
function toLocalTime(t: Date) {
  return localTime(t.getUTCHours(), t.getUTCMinutes());
}

function toAppointment(row: ApptRow) {
  return appointment(
    row.id,
    row.patientId,
    timeSlot(toLocalTime(row.startTime), toLocalTime(row.endTime)),
  );
}

/** Assemble the Agenda aggregate from its persisted parts. agenda() sorts internally. */
export function toAgenda(whRow: WhRow, apptRows: ApptRow[]): Agenda {
  return agenda(
    apptRows.map(toAppointment),
    workingHours(toLocalTime(whRow.openTime), toLocalTime(whRow.closeTime)),
  );
}
```

This is the ONLY file in the codebase that imports BOTH `@prisma/client` AND `../../../domain`. That
co-location is the hexagonal seam; it is the adapter's job to translate.

### `dto/plan.mapper.ts` — Plan → PlanResponseDto (the serialization gotcha)

`LocalTime` is a class with a private constructor and a `toString()` → `"HH:MM"`. **`JSON.stringify`
does NOT invoke `toString()` on nested class instances** — it would emit `{}` for each `LocalTime`,
silently producing `{ start: {}, end: {} }`. The mapper MUST call `.toString()` explicitly on every
`LocalTime`.

```ts
import type { Plan, PlanOperation, Conflict, TimeSlot } from '../../domain';
import type { PlanResponseDto, OperationDto, ConflictDto, TimeSlotDto } from './plan-response.dto';

function toSlotDto(slot: TimeSlot): TimeSlotDto {
  return { start: slot.start.toString(), end: slot.end.toString() }; // explicit HH:MM
}

function toOperationDto(op: PlanOperation): OperationDto {
  return {
    type: op.type,
    appointmentId: op.appointmentId,
    patientId: op.patientId,
    from: toSlotDto(op.from),
    to: toSlotDto(op.to),
  };
}

function toConflictDto(c: Conflict): ConflictDto {
  return { appointmentId: c.appointmentId, reason: c.reason, proposedSlot: toSlotDto(c.proposedSlot) };
}

export function mapPlanToDto(plan: Plan, confidence: number): PlanResponseDto {
  return {
    status: 'proposed',
    operations: plan.operations.map(toOperationDto),
    conflicts: plan.conflicts.map(toConflictDto),
    confidence,
  };
}
```

### `application/messages.service.ts` (new shape)

```ts
@Injectable()
export class MessagesService {
  constructor(
    @Inject(LANGUAGE_PORT) private readonly language: LanguagePort,
    @Inject(AGENDA_REPOSITORY) private readonly agendaRepo: AgendaRepository,
    @Inject(PLANNER_REGISTRY) private readonly registry: PlannerRegistry,
  ) {}

  async process(dto: MessageDto): Promise<PlanResponseDto> {
    const interpreted = await this.language.interprets(dto.message);
    const intent: Intent = {
      kind: interpreted.intent.kind,
      params: interpreted.intent.params,
    };

    const agenda = await this.agendaRepo.findAgendaForDate(DOCTOR_ID, today());
    if (agenda === null) {
      throw new UnprocessableEntityException({ error: 'agenda_not_found' });
    }

    const plan = recalculate(this.registry, agenda, intent);
    return mapPlanToDto(plan, interpreted.confidence);
  }
}
```

`PLANNER_REGISTRY` token lives in `application/messages.service.ts` (or a small `tokens.ts`). The
service does NOT import `buildPlannerRegistry` — the composition root builds it and injects the value,
keeping the service free of construction concerns.

### Test override pattern (no DB)

```ts
Test.createTestingModule({ imports: [AppModule] })
  .overrideProvider(LANGUAGE_PORT).useValue({ interprets: jest.fn().mockResolvedValue(mockIntentResponse) })
  .overrideProvider(AGENDA_REPOSITORY).useValue(new InMemoryAgendaRepository(seededAgenda))
  .compile();
```

Overriding `AGENDA_REPOSITORY` means `PrismaAgendaRepository`/`PrismaService` are never instantiated in
e2e, so **e2e runs with NO database**. `PrismaClient` is constructed lazily and only `$connect`s on
`onModuleInit`; because the provider is overridden, NestJS still registers `PrismaService` but nothing
calls its methods. If NestJS instantiates it, `$connect` is only triggered in `onModuleInit` against a
real connection attempt — to be safe, e2e ALSO overrides `PrismaService` with a no-op stub, OR
`PrismaService` is only referenced by `PrismaAgendaRepository` (which is overridden away). Preferred:
override `AGENDA_REPOSITORY`; if e2e logs a connect attempt, add `.overrideProvider(PrismaService).useValue({})`.

## 6. Constants & edge handling

### `DOCTOR_ID` and `today()`

Create `apps/scheduling/src/messages/application/scheduling.constants.ts`:

```ts
/** Single-doctor MVP — hardcoded composition-time constant (ADR/proposal decision 4). */
export const DOCTOR_ID = '00000000-0000-0000-0000-000000000001';

/** Server clock, date-only at request time (proposal decision 5). */
export function today(): Date {
  const now = new Date();
  return new Date(Date.UTC(now.getUTCFullYear(), now.getUTCMonth(), now.getUTCDate()));
}
```

`today()` returns UTC midnight so it matches the `@db.Date` column representation. The seeded
`working_hours.day` and `appointments.day` rows for tests/dev must use the same UTC-date convention.

### Agenda-not-found → 422

Thrown in **`MessagesService`** (not the controller). The service owns the orchestration decision; the
controller stays a thin pass-through. `UnprocessableEntityException({ error: 'agenda_not_found' })` →
HTTP 422. Rationale: the request is well-formed (passes `MessageDto` validation) but the system state
(no agenda for today) makes it unprocessable. 404 is wrong (the endpoint exists); a stub/empty agenda is
wrong (would silently produce an empty plan and hide a data-setup error).

### Empty agenda (working hours exist, zero appointments)

Valid and distinct from not-found. `toAgenda(whRow, [])` → `agenda([], workingHours(...))`. `recalculate`
runs, `DelayPlanner` produces `operations: [], conflicts: []`. Response is `200` with empty arrays. The
repository returns `null` ONLY when `working_hours` has no row for `(doctorId, day)`.

## 7. Prisma adapter (`prisma-agenda.repository.ts`)

```ts
@Injectable()
export class PrismaAgendaRepository implements AgendaRepository {
  constructor(private readonly prisma: PrismaService) {}

  async findAgendaForDate(doctorId: string, date: Date): Promise<Agenda | null> {
    const wh = await this.prisma.workingHours.findUnique({
      where: { doctorId_day: { doctorId, day: date } },
    });
    if (wh === null) return null;                 // no working hours → 422 upstream

    const appts = await this.prisma.appointment.findMany({
      where: { doctorId, day: date },
      orderBy: { startTime: 'asc' },
    });

    return toAgenda(wh, appts);                   // mapper owns row→domain
  }
}
```

`workingHours.findUnique` uses the `@@unique([doctorId, day])` compound key (`doctorId_day`). The adapter
contains the only `@prisma/client` query code in the app besides `PrismaService`.

## ADR-style decisions

**D1 — Mapper placement by foreign-type ownership.** `agenda.mapper` (imports Prisma rows) lives in
`infrastructure/prisma/`; `plan.mapper` (imports domain + emits DTO) lives in `dto/`. Rejected: a shared
`mappers/` dir — would invite a domain↔infra import shortcut and blur the seam.

**D2 — Exclusion constraint via raw SQL `tsrange(date+time)`.** Postgres lacks `timerange`; composing a
fixed date with `TIME` yields a `timestamp` for `tsrange(..., '[)')`, matching domain half-open overlap.
Rejected: `UNIQUE(doctor_id, day, start_time)` — only blocks identical starts, not true overlaps (fails ADR-0011).

**D3 — 422 thrown in the service, not the controller.** Orchestration owns the state decision; controller
stays thin. Rejected: controller-level guard — would duplicate domain knowledge of "what makes a request
unprocessable" into the transport layer.

**D4 — Integration test self-skips on missing `DATABASE_URL`.** Keeps `pnpm test` green without Postgres,
yet runs full DB + constraint validation when Neon is wired. Rejected: testcontainers — heavy devDep for an
MVP; Neon is remote and the user provisions it. Rejected: a separate jest project — `describe.skip` is simpler.

**D5 — `change_history` is fail-loud, not no-op.** A throwing stub surfaces any premature Phase-4 write
immediately. Rejected: silent no-op — would hide a contract violation until production.

**D6 — Drop the docker-compose Postgres.** Neon is remote; a local container is dead weight that misleads
`make dev`. Rejected: keep it "just in case" — implies a local DB Phase 2 never uses.

## Risks

- **LocalTime serialization (High):** mitigated by `plan.mapper` explicit `.toString()` + a dedicated
  `plan.mapper.spec.ts` asserting string output. Any future DTO field carrying a `LocalTime` must go
  through a mapper, never raw `JSON.stringify`.
